## 🎯 **解决的关键问题**

### **问题 1: getSBDBundles SOAP 格式**

**迭代次数**: 4 次  
**最终解决方案**:
```xml
<request>
    <fromBundleId>0</fromBundleId>       <!-- 查询参数 -->
    <forActivate>true</forActivate>      <!-- 查询参数 -->
    <sbdPlan />                          <!-- 服务类型标识 -->
</request>
```

**关键发现**:
- ❌ 不能用泛型 `<plan>`
- ❌ 不能直接放参数
- ❌ 不能把参数放在 `<sbdPlan>` 内
- ✅ 查询参数在外，`<sbdPlan />` 空标签

### **问题 2: accountSearch 字段解析**

**迭代次数**: 1 次  
**最终解决方案**:
```python
# 正确的字段名
account_elem = subscriber.find('.//accountNumber')  # ✅
# 不是 subscriberAccountNumber ❌
```

**关键发现**:
- WSDL 文档字段名可能不准确
- 需要遍历列表查找匹配的 IMEI
- 实际 API 返回结构与文档不同

### **问题 3: accountUpdate 缺少字段**

**迭代次数**: 1 次  
**最终解决方案**:
```xml
<sbdSubscriberAccount2>
    ...
    <newStatus>ACTIVE</newStatus>        <!-- 必需！ -->
    <plan>...</plan>
</sbdSubscriberAccount2>
```

**关键发现**:
- accountUpdate 同时处理费率和状态
- newStatus 是必需字段
- 不能只更新费率不指定状态

### **问题 4: PENDING 状态理解**

**迭代次数**: 完全理解  
**关键发现**:

1. **PENDING 是衍生状态**
   - 系统自动计算
   - 无法手动设置
   - 表示后台有未完成的订单

2. **PENDING 期间的限制**
   - 无法修改账号
   - 无法变更状态
   - 设备无法使用

3. **解除方式**
   - 自动转化（系统完成后）
   - 等待并重试
   - 联系 Tier 2 Support

## 💡 **关键学习**

### **1. WSDL 文档的局限性**

**发现**:
- WSDL 是指导，不是绝对真理
- 字段名可能不准确（subscriberAccountNumber vs accountNumber）
- 结构说明可能有歧义（plan vs sbdPlan）
- **实际 API 测试是唯一的真理**

**教训**:
- ✅ 总是用实际响应验证
- ✅ 不要过度依赖文档
- ✅ 错误消息是最好的指导

### **2. 迭代调试的价值**

**发现**:
- 每次失败都提供新线索
- 错误消息通常很准确
- 小步快跑比一次性大改更有效

**教训**:
- ✅ 接受失败是学习过程
- ✅ 详细记录每次尝试
- ✅ 从错误中提取信息

### **3. 状态机制的复杂性**

**发现**:
- PENDING 是衍生状态，有特殊规则
- 状态转换有顺序和限制
- 超时可能意味着成功（后台处理）

**教训**:
- ✅ 理解系统的状态模型
- ✅ 考虑异步处理
- ✅ 实现重试和等待逻辑

### **4. 错误消息分析**

**发现**:
- "Expected elements are..." 直接告诉正确结构
- "Field XXX required" 明确缺少什么
- "No updates while pending" 说明状态问题

**教训**:
- ✅ 仔细阅读错误消息
- ✅ 错误消息比文档更准确
- ✅ 综合多次错误理解全貌
