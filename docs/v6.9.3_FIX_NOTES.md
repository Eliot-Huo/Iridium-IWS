## 💡 **关键发现**

### **API 的真正要求**

从第一次错误消息看到 API 期望的元素：
```
Expected elements are:
- sbdPlan
- burstDevicePlan
- m2mPlan
- openPortPlan
- burstServicePlan
```

**这意味着**：
1. ❌ 不能用泛型的 `<plan>`
2. ❌ 不能直接放参数
3. ✅ **必须用具体的服务类型标签**（`<sbdPlan>`, `<m2mPlan>` 等）

**为什么？**
- API 通过 plan 对象的**类型**（sbdPlan, m2mPlan 等）来确定要返回哪些方案
- 如果没有指定类型，API 不知道要返回 SBD 方案还是其他类型的方案

## 🎓 **关键学习**

### **1. API 的细微要求**

WSDL 文档可能说：
- "使用 Plan 对象"

实际 API 要求：
- 使用**具体类型**的 plan 对象（`<sbdPlan>`, `<m2mPlan>` 等）
- 不是泛型的 `<plan>`

---

### **2. 错误信息的价值**

**第一次错误**：
```
Expected elements are: sbdPlan, burstDevicePlan, ...
```
→ 告诉我们需要用具体的类型标签

**第二次错误**：
```
No plan provided. Unable to determine service type
```
→ 确认 API 需要 plan 对象来确定服务类型

---

### **3. 迭代调试的重要性**

```
尝试 1 → 发现不能用 <plan>
尝试 2 → 发现不能直接放参数
尝试 3 → 理解需要 <sbdPlan>
```
